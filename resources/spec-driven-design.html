Basic Spec Driven Design with Antigravity, Agents-CLI and Gemini Enterprise

Learn how to design, build, and deploy high-quality AI agents starting with a specification-first methodology.

Introduction

Spec Driven Design (SDD) is a methodology where you define the goals, constraints, tool contracts, and success criteria of your AI Agent in a formal specification *before* writing code. Designing spec-first prevents agent scope creep, reduces model hallucinations, and aligns development directly with automated evaluations.

Why Spec-First?

Just as API-first development uses OpenAPI specs to align frontend and backend teams, Spec-Driven Agent Design uses agent specifications to align your agent code, prompt engineering, and LLM-as-a-judge evaluation criteria.

Workflow Overview

The standard development lifecycle when using the agents-cli and Antigravity comprises four distinct phases:

  1. Scaffold: Initialize the agent structure with predefined templates.
  2. Build: Code the custom tools and define orchestration behaviors.
  3. Evaluate: Systematically test agent outputs using the Quality Flywheel.
  4. Deploy & Publish: Host the agent on Cloud Run or Agent Runtime and make it discoverable in Gemini Enterprise.

1. Scaffold your Project

To start a new project, use the agents-cli scaffold create command. This command sets up the necessary directory structures, configuration manifests, and basic evaluation dataset files.

# Install agents-cli tool
uv tool install google-agents-cli

# Scaffold a new Agent project
agents-cli scaffold create my-first-agent
Enhance Existing Repos

If you already have agent code files, run agents-cli scaffold enhance . to inject the standard config, evaluation runners, and CI/CD templates directly into your existing project.

2. Implement Agent Logic

Write your agent logic using the Agent Development Kit (ADK) Python SDK. In your main agent file (usually agent.py), define your agent instance, configure the model, and register tools.

from adk import Agent, Tool

# Define a custom tool
def get_current_weather(location: str) -> str:
    return f"The weather in {location} is sunny, 72°F."

weather_tool = Tool(
    name="get_weather",
    func=get_current_weather,
    description="Get weather updates for a specific location."
)

# Create the Agent
my_agent = Agent(
    name="WeatherAgent",
    instruction="You are an assistant. Use tools to provide weather info.",
    tools=[weather_tool],
    model="gemini-2.5-flash"
)

Quick Smoke Testing

Instead of writing unit tests that make brittle assertions on natural language output, perform smoke tests directly in the terminal:

agents-cli run "What's the weather in Seattle?"

3. Run Evaluations (The Quality Flywheel)

Evaluation is the key to creating robust agent behaviors. agents-cli uses an LLM-as-a-judge system to grade agent outputs against concrete criteria defined in your dataset.

Avoid Brittle Pytests

Never write standard pytests that check if the model's text response matches specific strings. Natural language is non-deterministic. Use the evaluation dataset framework instead.

Execute evaluation runs by executing:

agents-cli eval run

This runs the two underlying phases consecutively:

  • Generate: Sends your dataset prompts to the agent and records the execution traces.
  • Grade: Runs judges against those traces to evaluate guidelines, accuracy, and tool usage.

4. Deploy & Publish to Gemini Enterprise

When evaluations pass your target threshold, deploy your agent. The Agent Platform supports multiple deployment targets including Agent Runtime and Cloud Run.

agents-cli deploy

Registering the Agent

After deployment, make the agent available in the Gemini Enterprise App. This registers it in the Agent Registry so other enterprise users (or other agents) can interact with it.

agents-cli publish gemini-enterprise

Best Practices

  • Isolate logic: Keep tools focused on simple input/output tasks.
  • Iterate on Eval cases: Start with 2-3 simple evaluations and gradually expand to edge cases as you resolve bugs.
  • Never change models arbitrarily: Keep models consistent with your evaluation environment to maintain predictable behaviors.
